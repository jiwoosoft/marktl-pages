harness_eng 비교 평가를 위한 공개용 리서치 리포트

초록

원문 노트는 harness_eng_with와 harness_eng_without 두 폴더를 비교 평가하고, 결과를 comparison_report.md로 작성하라는 실행 지시다. 핵심은 단순 감상이 아니라 파일 근거, 실제 검증 결과, 치명적 결함, 최종 판정을 포함한 평가 보고서를 만드는 데 있다.

평가 프레임은 기능 완성도부터 릴리즈 준비도까지 총 100점으로 구성되어 있으며, 구현물의 제품성, 데이터 신뢰도, 코드 지속 가능성, 검증 가능성을 균형 있게 판단하도록 설계되어 있다.

핵심 요약

Finding 01 두 폴더는 동일한 기준표로 점수화되어야 하며, 각 항목은 반드시 파일 단위 근거를 요구한다.
Finding 02 build, test, validation script 실행 여부가 평가 신뢰도를 좌우하는 핵심 증거로 지정되어 있다.
Finding 03 최종 산출물은 단순 비교표가 아니라 치명적 결함과 추가 검증 권장 사항까지 포함한 의사결정 문서다.
“좋은 비교 평가는 어느 쪽이 더 나은지 말하는 데서 끝나지 않는다. 왜 그런지, 어떤 파일이 그 판단을 뒷받침하는지까지 보여준다.”

평가 프레임

원문은 총 7개 항목, 100점 만점의 평가 체계를 제시한다. 이 체계는 기능, 데이터, 사용자 경험, 코드 품질, 검증, 문서화, 릴리즈 준비도를 함께 본다는 점에서 실무형 리뷰에 가깝다.

평가 항목 배점 판단 초점
기능 완성도 20점 요구 기능이 실제로 구현되어 있고 실행 가능한지 평가한다.
데이터 품질 및 정규화 품질 20점 데이터 구조, 중복 제거, 정규화, 품질 관리 방식이 충분한지 확인한다.
UI/UX 사용성 15점 사용 흐름, 화면 구조, 입력과 출력의 명료성, 오류 경험을 살핀다.
코드 구조와 유지보수성 15점 모듈 경계, 중복, 의존성, 변경 용이성을 기준으로 본다.
검증 체계 15점 build, test, validation script가 존재하고 실제로 작동하는지 평가한다.
문서화 10점 README, 실행법, 갱신법, 운영 절차가 재현 가능하게 정리되어 있는지 본다.
릴리즈 준비도 5점 배포 전 점검, 설정, 산출물 정리, 사용 가능 상태를 확인한다.

권장 평가 방법

비교 대상이 되는 두 폴더는 harness_eng_with와 harness_eng_without이다. 평가자는 먼저 저장소의 CLAUDE.md를 읽고, 그 안에 정의된 평가 원칙을 우선 적용해야 한다.

CLAUDE.md 확인
→
두 폴더 파일 근거 수집
→
점수화 및 최종 판정

증거 기반 점수화

각 점수는 추정이 아니라 파일 경로, 구현 내용, 문서, 스크립트 실행 결과로 설명되어야 한다. “좋아 보인다”보다 “어떤 파일의 어떤 구조 때문에 점수를 부여했다”가 중요하다.

실행 결과 반영

가능하면 실제 build, test, validation을 실행하고 결과를 반영한다. 실행하지 못한 경우에도 그 이유와 남는 리스크를 명시해야 한다.

원문 지시의 구조

아래는 원문 노트에 포함된 핵심 지시를 보고서 작성 기준으로 재배열한 것이다.

CLAUDE.md를 먼저 읽고 평가 원칙을 따르라. harness_eng_with 와 harness_eng_without 두 폴더를 비교 평가하라. 평가 기준: 1. 기능 완성도 (20점) 2. 데이터 품질 및 정규화 품질 (20점) 3. UI/UX 사용성 (15점) 4. 코드 구조와 유지보수성 (15점) 5. 검증 체계(build/test/validation script) (15점) 6. 문서화(README, 실행법, 갱신법) (10점) 7. 릴리즈 준비도 (5점)
출력 형식: 1. 총점 비교표 2. 항목별 비교 3. 치명적 결함 목록 4. 최종 판정 5. 추가 검증 권장 사항 최종 결과를 comparison_report.md 파일로 작성하라.

보고서 산출물의 권장 구성

최종 보고서인 comparison_report.md는 평가자가 바로 의사결정에 사용할 수 있도록 다음 순서를 따르는 것이 적합하다.

  1. 총점 비교표. 두 폴더의 총점과 항목별 점수를 한눈에 비교한다.
  2. 항목별 비교. 각 항목마다 점수 이유와 파일 근거를 병기한다.
  3. 치명적 결함 목록. 실행 불가, 데이터 손상, 검증 부재 같은 중대 리스크를 별도로 표시한다.
  4. 최종 판정. 어느 구현이 더 우수한지 명확히 결론 내린다.
  5. 추가 검증 권장 사항. 남은 불확실성과 다음 확인 작업을 제안한다.
공개용 리포트에서는 “승자”만 강조하기보다, 어떤 조건에서 그 결론이 성립하는지 함께 적는 편이 독자에게 더 유용하다.

시사점

이 평가 지시문은 코드 리뷰와 제품 평가의 경계에 있다. 단순히 소스가 존재하는지 확인하는 데서 멈추지 않고, 데이터 품질, 사용자 경험, 검증 체계, 문서화까지 함께 판단하므로 실제 인수 검토나 릴리즈 판단에 사용할 수 있다.

특히 “파일 근거”와 “실제 실행 결과”를 요구한다는 점이 중요하다. 이는 평가자의 주관을 줄이고, 다음 작업자가 같은 저장소를 열었을 때 판단을 재현할 수 있게 만든다.
