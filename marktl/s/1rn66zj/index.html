Abstract

이 리포트는 Discord를 단순한 채팅 도구가 아니라 AI 에이전트 운영실로 재구성하는 방법을 다룬다. 핵심은 OpenClaw를 중심에 두고 Codex와 Claude Agent를 같은 작업 공간에 배치해, 사람이 모든 중간 판단을 직접 처리하지 않아도 계획, 구현, 리뷰, 수정, 검증이 순환하도록 만드는 것이다. 영상의 사례는 완전 자동화라기보다 “사람의 병목을 줄이는 반자동 협업 시스템”에 가깝다.

핵심 발견

이 셋업의 가치는 특정 봇 하나를 붙이는 데 있지 않다. 진짜 변화는 작업 단위, 역할, 리뷰 흐름, 세션 관리를 Discord라는 공용 인터페이스 안에 넣는 데 있다.

1. Discord는 멀티프로젝트 작업실이 된다

채널은 프로젝트 단위, thread는 작업 단위로 쓰인다. 텔레그램식 단일 대화창보다 문맥 오염을 줄이고 병렬 작업을 열기 쉽다.

2. OpenClaw는 오케스트레이터 역할을 맡는다

사용자의 지시를 받고, 작업을 정리하고, Codex 기반 구현을 진행하며, 필요할 때 Claude Agent를 호출한다.

3. Claude는 무조건적인 정답자가 아니라 리뷰어다

구현 계획, 코드 변경, 문서, UX 판단을 검토한다. 다만 Codex가 Claude의 피드백을 그대로 수용하지 않도록 비판적 검토 절차가 필요하다.

4. 컨텍스트 관리는 운영 품질을 좌우한다

OpenClaw와 Claude Bot의 세션은 별개다. 한쪽 컨텍스트만 정리하면 다른 쪽이 여전히 느려지거나 혼란스러울 수 있다.

작업실의 기본 아키텍처

영상에서 제안하는 구조는 “사람이 하나의 AI에게 명령한다”보다 한 단계 더 나아간다. 사용자는 Discord에서 작업을 요청하고, OpenClaw가 전체 흐름을 조율하며, Codex가 구현하고, Claude Agent가 리뷰한다. 이후 Codex는 리뷰를 검토하고 필요한 부분만 반영한다.

그림 1. Discord thread 안에서 반복되는 구현과 리뷰의 기본 루프.

구성 요소 역할 운영 포인트
Discord 서버 프로젝트별 채널과 작업별 thread를 담는 운영 공간 채널은 큰 주제, thread는 개별 작업으로 나누는 것이 핵심이다.
OpenClaw Bot 사용자의 명령을 받고 Codex 기반 작업을 실행하는 오케스트레이터 voice note나 일반 메시지에도 반응하게 두면 모바일 지시가 쉬워진다.
Claude Bot 계획과 구현을 검토하는 리뷰어 required mention을 켜서 직접 호출될 때만 반응하게 해야 한다.
Mac mini 또는 로컬 호스트 Claude CLI, Claude Code, Node.js 중계 프로그램이 실행되는 기반 봇은 Discord에 있지만 실제 작업 능력은 로컬 실행 환경과 연결된다.
좋은 멀티에이전트 셋업은 더 많은 AI를 붙이는 것이 아니라, 누가 언제 말하고 언제 멈춰야 하는지를 정하는 운영 규칙에서 시작된다.

왜 Telegram보다 Discord인가

Telegram은 빠르게 붙이고 테스트하기 좋지만 하나의 대화창에 여러 프로젝트와 지시가 섞이기 쉽다. 반면 Discord는 채널과 thread를 기본 구조로 제공하기 때문에 프로젝트, 작업, 리뷰 로그를 분리할 수 있다.

채널은 프로젝트, thread는 작업

예를 들어 영상 제작 채널, 리서치 채널, 앱 개발 채널, 개인 자동화 채널을 따로 만들 수 있다. 그 안에서 “버튼 버그 수정”, “SEO 개선”, “목업 페이지 구현” 같은 작업을 각각 thread로 나누면, 에이전트가 들고 가야 할 문맥이 훨씬 작아진다.

메신저의 알림성이 운영 인터페이스가 된다

Discord는 작업 완료, 리뷰 요청, 판단 대기 상태를 알림으로 받기 쉽다. 사용자는 랩탑 앞에 계속 앉아 있지 않아도 모바일에서 방향을 정하거나 다음 작업을 줄 수 있다.

비교 항목 Telegram 중심 운영 Discord 중심 운영
문맥 분리 단일 대화창에 섞이기 쉽다 채널과 thread로 자연스럽게 분리된다
병렬 작업 한 흐름이 끝나기를 기다리기 쉽다 여러 thread에서 동시에 진행하기 쉽다
리뷰 가시성 로그가 한 줄로 섞인다 작업별 리뷰 맥락을 따라가기 쉽다
운영 확장성 간단한 개인 비서에 적합하다 작업실과 운영 시스템에 가깝다

AI끼리 리뷰하게 만들 때의 함정

멀티에이전트 워크플로의 장점은 한 모델이 놓친 부분을 다른 모델이 잡아낼 수 있다는 데 있다. 하지만 단순히 “Claude의 피드백을 반영해”라고 지시하면 Codex가 모든 리뷰를 무비판적으로 받아들이는 문제가 생길 수 있다.

운영 원칙

리뷰 피드백은 항목별로 나누고, 각 항목에 대해 동의, 비동의, 부분 동의를 먼저 판단한 뒤, 근거가 확인된 항목만 구현 계획에 반영해야 한다.

비판적 리뷰 프롬프트의 골격

  • Claude의 리뷰를 항목별로 분해한다.
  • 각 항목에 대해 동의, 비동의, 부분 동의를 표시한다.
  • 목표, 제약 조건, 구현 리스크, 사용자 경험 기준으로 검토한다.
  • 수용할 항목만 계획에 반영한다.
  • 이견이 있는 항목은 다시 Claude에게 보내 합의될 때까지 논의한다.
  • 중요 쟁점이 정리된 뒤에 구현으로 넘어간다.

이 절차가 없으면 겉보기에는 협업이 매끄럽지만 실제로는 한쪽 에이전트가 다른 에이전트의 의견을 그대로 따르는 구조가 된다. 사람 사이의 코드 리뷰와 마찬가지로, 리뷰는 권위가 아니라 검토 대상이다.

컨텍스트와 세션 관리가 성패를 가른다

멀티에이전트 운영에서 가장 쉽게 과소평가되는 부분은 컨텍스트 관리다. Discord thread가 남아 있다고 해서 에이전트의 세션 컨텍스트가 그대로 살아 있는 것은 아니다. 오래된 thread를 다시 열었다면 먼저 과거 로그를 읽고 작업 상태를 재구성하도록 요청해야 한다.

새 작업은 새 thread에서 시작한다

한 채널에서 모든 작업을 이어가면 컨텍스트가 쌓이고, 모델은 매번 더 많은 과거 대화를 끌고 간다. 작은 작업이라도 thread를 열고 완료되면 닫거나 정리하는 편이 안정적이다.

OpenClaw와 Claude Bot의 세션은 별개다

OpenClaw의 /status가 괜찮아 보여도 Claude Bot의 컨텍스트가 이미 무거워져 있을 수 있다. 반대로 Claude 쪽을 새로 고쳐도 OpenClaw의 작업 세션이 혼잡할 수 있다. 따라서 운영자는 두 에이전트의 컨텍스트를 모두 점검해야 한다.

상황 위험 권장 대응
thread가 오래 열려 있음 응답 지연, 토큰 낭비, 맥락 혼란 상태 요약 후 새 thread로 이전한다.
아카이브된 thread 재개 메시지는 보이지만 에이전트 세션은 비어 있을 수 있음 지난 로그를 읽고 작업 상태를 복원하게 한다.
Claude Bot이 반응하지 않음 권한 문제로 오해하기 쉬움 gateway 재시작과 mention 설정을 먼저 확인한다.
여러 채널을 동시에 운영 위험한 세션을 사람이 놓치기 쉬움 마스터 채널에서 세션 상태 리포트를 정기적으로 생성한다.
멀티에이전트 생산성은 모델 성능만으로 결정되지 않는다. 오래 달리는 시스템일수록 환기, 요약, 재수화가 운영의 일부가 된다.

실무적 시사점

이 영상이 보여주는 방향은 “완전 자율 개발자”보다 현실적이다. 에이전트들이 밤새 전부 완성해 주는 세계라기보다, 사람이 잠시 빠져 있어도 계획과 리뷰와 구현이 어느 정도 앞으로 나아가는 운영 구조다.

사람의 역할은 사라지지 않고 바뀐다

사람은 모든 리뷰의 병목이 되는 대신, 에이전트가 합의하지 못한 결정, 취향과 목적이 중요한 판단, 제품 방향과 리스크가 걸린 선택을 맡는다. 단순 반복 검토는 에이전트에게 넘기고, 인간 판단이 필요한 지점에서만 개입하는 방식이다.

작게 시작하는 것이 중요하다

처음부터 거대한 자동화 시스템을 만들 필요는 없다. 레스토랑 예약, 아침 브리핑, 작은 코드 수정, 문서 리뷰처럼 반복되는 일의 일부를 위임하면서 지시 방식과 에이전트의 한계를 익히는 것이 현실적인 출발점이다.

  • 먼저 Discord 서버와 프로젝트별 채널 구조를 만든다.
  • OpenClaw Bot과 Claude Bot을 분리해 초대한다.
  • Claude Bot에는 required mention을 켜서 불렸을 때만 반응하게 한다.
  • OpenClaw에는 일반 메시지와 voice note 기반 지시를 허용한다.
  • 작업마다 새 thread를 열고 완료 후 요약하거나 닫는다.
  • 리뷰 피드백은 비판적으로 수용하도록 프롬프트와 메모리에 규칙을 저장한다.

결론은 단순하다

AI 에이전트가 많아질수록 중요한 것은 더 강한 명령어가 아니라 더 좋은 운영 구조다. Discord를 프로젝트별 작업실로 쓰고, OpenClaw와 Codex와 Claude의 역할을 분리하고, 리뷰를 비판적으로 순환시키면 사람은 모든 중간 판단의 병목에서 벗어나 더 중요한 결정에 집중할 수 있다.
